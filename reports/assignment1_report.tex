\section{Executive Scope}
The modeled process covers an e-purchase from order submission through validation, stock handling, payment, fulfillment, delivery, complaint handling, and final archiving. To maximize assignment coverage, the same process is documented in both required modeling styles:
\begin{itemize}
  \item BPMN 2.0 models for the as-is and to-be business process.
  \item Petri-net models for the same as-is and to-be behavior, including choices, synchronization, loops, and final markings.
\end{itemize}

The to-be design improves early validation, parallelizes independent checks, connects complaints to the same order case, and adds proactive SLA monitoring for delayed goods.

\subsection{As-Is BPMN}
The as-is process starts when the order is received. The system captures the order and checks stock. If stock is unavailable, the order is rejected. If stock is available, the item is reserved, address data is checked, and missing address data is requested. Payment authorization and warehouse fulfillment then run in parallel. After invoice emission and shipment, the process waits for a delivery event.

The delayed-goods scenario is represented by an intermediate timer event after shipment. If delivery confirmation is not received within the SLA, the timer branch registers a complaint, escalates the courier, and loops back to the delivery wait state.

\WideFigure{reports/svg-cache/as_is_bpmn.pdf}{As-is BPMN 2.0 model exported from the BPMN source as SVG and rendered into the report cache.}

\subsection{To-Be BPMN}
The to-be process moves validation earlier, runs stock and payment checks in parallel, standardizes fulfillment preparation, adds tracking, and makes the complaint path proactive. The SLA breach event opens a complaint automatically, notifies the customer, expedites the courier, and then keeps the order under tracking until delivery confirmation is received.

\section{Petri-Net Model}
The Petri-net model is the second modeling method. States are represented as places, activities and routing events as transitions, and allowed movement as directed arcs. One token starts in the initial place. Silent transitions are used only for routing choices, parallel splits, and joins.

\subsection{As-Is Petri Net}
The as-is Petri net has one token in \texttt{p\_start}. The stock decision place enables either rejection or reservation. The preparation split produces one token for the payment branch and one token for the fulfillment branch; the join requires both invoice completion and shipment before delivery waiting can continue. The goods-delay timer transition creates the complaint token, and the escalation transition loops back to the delivery waiting place.

\WideFigure{reports/svg-cache/as_is_petri_net.pdf}{As-is Petri net exported from the DOT/PNML model as SVG and rendered into the report cache.}

\subsection{To-Be Petri Net}
The to-be Petri net shows two improvements formally. First, stock check and payment authorization are enabled together after validation, then synchronized before the order validity decision. Second, invoice emission and physical fulfillment are split into parallel branches and joined before tracking. The SLA breach transition sends the case through complaint opening, customer notification, and courier expedition before returning to the SLA wait place.

\WideFigure{reports/svg-cache/to_be_petri_net.pdf}{Enhanced to-be Petri net showing validation, parallel checks, fulfillment synchronization, and the complaint loop.}

\section{Case Coverage}
\begin{itemize}
  \item Normal order: checkout, validation, stock/payment checks, fulfillment, delivery confirmation, and archive.
  \item Rejected order: invalid payment, invalid data, or unavailable stock leads to rejection.
  \item Missing address: handled before stock is blocked in the to-be model.
  \item On-time delivery: delivery message event continues to archive.
  \item Goods delayed: timer/SLA event opens complaint, notifies customer, escalates courier, and loops back to delivery tracking.
  \item Audit trail: status updates and archiving support later monitoring and process mining.
\end{itemize}

\section{Conclusion}
The BPMN models provide the business-readable view required for process discussion, while the Petri nets provide the formal execution view required for reasoning about choices, concurrency, synchronization, and loops. The to-be model directly addresses the main weaknesses in the as-is process: late validation, reactive complaint handling, and weak status visibility.
