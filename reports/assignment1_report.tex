\section{Executive Scope}
The modeled process covers an e-purchase from order submission through validation, stock handling, payment, fulfillment, delivery, complaint handling, and final archiving. To maximize assignment coverage, the same process is documented in both required modeling styles:
\begin{itemize}
  \item BPMN 2.0 models for the as-is and to-be business process.
  \item Petri-net models for the same as-is and to-be behavior, including choices, synchronization, loops, and final markings.
\end{itemize}

The to-be design improves early validation, parallelizes independent checks, connects complaints to the same order case, and adds proactive SLA monitoring for delayed goods.

\subsection{As-Is BPMN}
The as-is process starts when the order is received. The system captures the order and checks stock. If stock is unavailable, the order is rejected. If stock is available, the item is reserved, address data is checked, and missing address data is requested. The two address outcomes meet at a confirmation task before the e-commerce pool splits into parallel handoff branches. Each branch uses explicit BPMN send/receive tasks: payment request then payment status, and fulfillment request then shipment status. Dashed message flows connect those handoff tasks to the external Payment Service, Warehouse/Fulfillment, and Courier/Shipment pools. After both statuses synchronize, the process monitors delivery status.

The delayed-goods scenario is represented by an intermediate timer event after shipment. If delivery confirmation is not received within the SLA, the timer branch registers a complaint, escalates the courier, and loops back to delivery monitoring before the next delivery-status decision. The delivery confirmation event remains a single catch event owned by the delivery wait.

\subsection{To-Be BPMN}
The to-be process moves validation earlier, runs stock and payment checks in parallel, reviews the combined check results, standardizes fulfillment preparation, adds explicit payment receipt, adds shipment tracking, and makes the complaint path proactive. The early check section uses explicit e-commerce send/receive tasks for payment pre-check and stock status, which fill the marketplace lane between the automated-check split and the checks-done join. The fulfillment section uses explicit e-commerce send/receive handoff tasks for invoice/payment status and label/shipment status, so the collaboration shows both internal sequence flow and straight cross-pool message flow. The SLA breach event opens a complaint automatically, notifies the customer, expedites the courier, and then loops back to shipment tracking before the next SLA-status decision.

\section{Petri-Net Model}
The Petri-net model is the second modeling method. States are represented as places, activities and routing decisions as transitions, and allowed movement as directed arcs. One token starts in the initial place. Routing choices, parallel splits, and joins are shown as normal transitions using the same visual style as the other transitions.

\subsection{As-Is Petri Net}
The as-is Petri net has one token in \texttt{p\_start}. The stock decision place enables either rejection or reservation. Rejection ends at the visible \texttt{p\_rejected} place. Both address outcomes produce the same preparation place, matching the corrected BPMN confirmation step. The preparation split produces one token for the payment branch and one token for the fulfillment branch; the join requires payment receipt and shipment before delivery waiting can continue. The goods-delay timer transition creates the complaint token, and the escalation transition loops back to the delivery waiting place.

\WideFigure{reports/svg-cache/as_is_petri_net.pdf}{As-is Petri net exported from the DOT/PNML model as SVG and rendered into the report cache.}

\subsection{To-Be Petri Net}
The to-be Petri net shows two improvements formally. First, stock check and payment authorization are enabled together after validation, then synchronized before the order validity decision; rejected orders end at \texttt{p\_rejected}. Second, invoice/payment receipt and physical fulfillment are split into parallel branches and joined before tracking. The SLA breach transition sends the case through complaint opening, customer notification, and courier expedition before returning to the SLA wait place.

\section{Case Coverage}
\begin{itemize}
  \item Normal order: checkout, validation, stock/payment checks, fulfillment, delivery confirmation, and archive.
  \item Rejected order: invalid payment, invalid data, or unavailable stock leads to rejection.
  \item Missing address: handled before stock is blocked in the to-be model.
  \item On-time delivery: delivery message event continues to archive.
  \item Goods delayed: timer/SLA event opens complaint, notifies customer, escalates courier, and loops back to delivery tracking.
  \item Audit trail: status updates and archiving support later monitoring and process mining.
\end{itemize}

\section{Conclusion}
The BPMN models provide the business-readable view required for process discussion, while the Petri nets provide the formal execution view required for reasoning about choices, concurrency, synchronization, and loops. The to-be model directly addresses the main weaknesses in the as-is process: late validation, reactive complaint handling, and weak status visibility.
