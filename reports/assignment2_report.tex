\section{Lecture Concepts and Algorithm}
The process-mining lecture defines the minimum event-log fields as case IDs and task IDs, with optional timestamp, resource, and data fields. This assignment uses case ID, timestamp, activity, and resource.

The Alpha relations used in the computation are:
\begin{itemize}
  \item Direct succession: $x > y$ when $x$ is directly followed by $y$ in at least one case.
  \item Causality: $x \rightarrow y$ when $x > y$ and not $y > x$.
  \item Parallelism: $x \parallel y$ when both $x > y$ and $y > x$.
  \item Choice: $x \# y$ when neither $x > y$ nor $y > x$.
\end{itemize}

The implemented Alpha algorithm follows these steps:
\begin{enumerate}
  \item Build $T_W$, the set of all activities in the workflow log.
  \item Build $T_I$, the activities that appear first in at least one trace.
  \item Build $T_O$, the activities that appear last in at least one trace.
  \item Compute direct succession, causality, parallelism, and choice.
  \item Build candidate pairs $(A,B)$ where every $a \in A$ causes every $b \in B$, activities inside $A$ are pairwise choice, and activities inside $B$ are pairwise choice.
  \item Keep only maximal pairs $Y_W$ and convert them into Alpha places.
  \item Add source and sink places to produce the mined workflow net.
\end{enumerate}

\section{Mined Process Interpretation}
The mined process has one start activity, \textit{Check stock availability}, and one end activity, \textit{Archive order}. The exact computations show 4 cases, 36 events, 12 activities, 18 observed direct-succession pairs, 14 causal pairs, 2 unordered parallel pairs, and 11 maximal Alpha places.

The behavior discovered from the traces is:
\begin{itemize}
  \item After stock checking, the process chooses between retrieving an available warehouse item and entering the materials/manufacturing branch.
  \item In the manufacturing branch, checking materials may lead directly to manufacturing or to requesting and obtaining raw materials first.
  \item Warehouse retrieval and manufacturing converge at confirming the order.
  \item Invoice/address order differs across traces, so the Alpha algorithm detects \textit{Emit invoice} and \textit{Get shipping address} as parallel.
  \item Payment/shipment order also differs across traces, so \textit{Receive payment} and \textit{Ship product} are parallel.
  \item Archiving is possible only after payment and shipment completion evidence exists in the log.
\end{itemize}

The log does not show a repeated-activity loop inside a case, so the mined model focuses on choices and concurrency rather than loop behavior. The main anomaly is the Case 3 event-ID prefix mismatch documented in preprocessing; it is treated as a data-quality issue rather than a behavioral exception because the activity names remain consistent with the overall process.

The PERT dependencies are the causal pairs from the Alpha calculation. For timing, each causal edge uses observed elapsed hours from the event log: optimistic is the minimum, most-likely is the median, pessimistic is the maximum, and expected duration is computed with $E=(O+4M+P)/6$. The critical path found by the automation is shown in orange in the SVG.

\section{Automation Design}
The automation script is \path{tools/process_mining_alpha.py}. It performs the following reusable workflow:
\begin{enumerate}
  \item Read any CSV containing \texttt{case\_id}, \texttt{timestamp}, \texttt{activity}, and \texttt{resource}.
  \item Sort events per case by timestamp.
  \item Extract traces, start activities $T_I$, end activities $T_O$, and task set $T_W$.
  \item Count direct succession pairs.
  \item Derive causality, parallel, and choice using the lecture definitions.
  \item Build maximal Alpha places $(A,B)$.
  \item Compute case cycle times and average waiting times between directly successive activities.
  \item Generate documentation outputs and online-visualization files.
\end{enumerate}

Run command:
{\small
\begin{verbatim}
python3 tools/process_mining_alpha.py \
  --input data/event_log.csv \
  --output-dir output
\end{verbatim}
}

\section{Timing Findings and Improvements}
The average observed cycle time is 5.28 days. The longest average waits are \textit{Ship product} to \textit{Receive payment} at 76.28 hours, \textit{Emit invoice} to \textit{Receive payment} at 57.32 hours, and \textit{Get shipping address} to \textit{Emit invoice} at 46.76 hours. These delays are waiting periods between handoffs, not the short system tasks themselves.

Recommended improvements are:
\begin{itemize}
  \item Integrate payment and shipment status so archive readiness is visible immediately.
  \item Add alerts for long gaps after invoice emission or shipment dispatch.
  \item Standardize whether payment must precede shipment or whether both may proceed independently; the log shows both orders.
  \item Monitor resource handoffs between SYS1, SYS2, employees, and DMS because handoffs are common sources of delay and error.
\end{itemize}
